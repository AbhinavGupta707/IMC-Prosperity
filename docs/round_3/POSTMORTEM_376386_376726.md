# Postmortem: R3 submissions 376386 and 376726

**Status:** Fixed. New bundle at `outputs/submissions/submission_r3_v2_fixed.py` (67KB).

## What happened

Two R3 submissions were made; both lost (0 and −$23,530). Both failures were **bundler bugs**, not strategy bugs. The strategies are correct; the flattening of source modules into a single-file Prosperity submission dropped or clobbered essential code.

## Root causes

### Submission 376386 (unminified bundle): crashed every tick → +0 P&L

The bundle dropped the body of `if self.orchestrator is not None:` in `Trader._run_body()`. Lines that build the portfolio snapshot and call `self.orchestrator.step(...)` were missing. `orch_summary` stayed `None`, then the subsequent `for product, orders in orch_summary.orders_by_product.items():` loop crashed with `AttributeError: 'NoneType' object has no attribute 'orders_by_product'`.

**Trader's outer `try/except` caught the crash and returned empty orders.** Every tick. 1000 lambda errors, 0 trades, +0 P&L.

**Why it happened:** this bundle was built WITHOUT the `--minify` flag, and the export path for the unminified case has different line-collapsing logic than the minified one. That code path was never tested end-to-end.

### Submission 376726 (minified bundle): wrong-symbol orders → −$23K

The minifier flattens every R3 module into one flat namespace. Module-level private constants (`_FOO`) with the **same name across modules overwrite each other** in global scope.

| Constant | Assignments across modules | Final value |
|---|---|---|
| `_POSITION_LIMIT` | 5 (HYDROGEL=200, VEV_4000=300, voucher=300, VELVET=200, lottery=300) | **300** |
| `_DEFAULT_PARAMS` | 2 (HYDROGEL params, VEV_4000 params) | **VEV_4000 params** |
| `_PRODUCT` | 2 ('VEV_4000', VELVETFRUIT_EXTRACT) | **VELVETFRUIT_EXTRACT** |
| `_IDLE_DELTA_THRESHOLD`, `_IDLE_POS_THRESHOLD` | minifier removed unused names | **Missing entirely** |

**Concrete damage:**

1. `vev4000_orders()` internally calls `take_clear_make(product=_PRODUCT, ...)`. At call time `_PRODUCT = 'VELVETFRUIT_EXTRACT'` (overwritten by velvet_hedge module). Result: **every VEV_4000 order was submitted with symbol=VELVETFRUIT_EXTRACT**. A "sell VEV_4000 at 1253" became "sell VELVET at 1253" — which immediately crossed VELVET's bid at 5265 and filled at 5265 with no hedge.

2. `velvet_hedge_orders()` relied on `_IDLE_DELTA_THRESHOLD` / `_IDLE_POS_THRESHOLD` to skip trading when `net_delta=0` and `velvet_position=0`. Those constants were stripped from the bundle, so the idle guard line evaluated `if abs_delta < _IDLE_DELTA_THRESHOLD and ...:` against `NameError` (which we... wait no, they were actually just absent — the guard logic was silently dropped during minification dead-code elimination).

3. Cumulative effect: every tick, aggressive-sell VELVET at the bid. Position hit −200 limit quickly. Cross-mode kicked in to cover. Round trip loss: ~5 ticks per cycle × ~500 cycles = ~$12,500, plus warehousing drag and unfilled-buy inefficiency = ~$23,530.

**Why it happened:** the minifier uses `python_minifier` with `rename_locals=False` — globals aren't renamed either. Same-named module privates are assumed not to collide. This assumption was never validated.

**Why HYDROGEL didn't trade either:** unclear. Our local replay (with correct module namespacing) shows HYDROGEL producing ~2 orders/tick that should fill. Best hypothesis: the prosperity platform's passive fill model is stricter than our local `passive_allocation=0.05` calibration, so quotes sitting one tick inside the deep wall (at 10002 bid / 10021 ask while market is 10003/10019) almost never get hit. The true edge from HYDROGEL MM at that quote placement may be much smaller than LOO suggested. Needs fresh submission with correct bundle to measure.

## Fixes applied

### 1. Rename all collision-prone module-private constants

All underscore-prefixed module constants in `src/strategies/round_3/*.py` are now module-prefixed:

| Before | After |
|---|---|

### 2. Bundle regression test suite

New: [tests/test_submission_bundle_r3.py](../../tests/test_submission_bundle_r3.py) — 13 tests that build the actual R3 bundle and assert:

**Static checks (source analysis):**
- Bundle contains `self.orchestrator.step(` (catches 376386-class bugs).
- Bundle contains `self.orchestrator.restore(`.
- **Zero bare occurrences of `_POSITION_LIMIT`, `_DEFAULT_PARAMS`, `_PRODUCT`, `_SOFT_CAP`, `_HARD_CAP`, `_TIGHT_BAND`, `_IDLE_*`, `_PARAMS_*`, `_LOTTERY_*`, `_PROBE_*`, `_VOUCHER_SPECS`** at module scope (catches 376726-class bugs).
- Required renamed constants (`_HYDROGEL_PARAMS`, `_VEV4000_PRODUCT`, `_VELVET_HEDGE_IDLE_DELTA`, etc.) ARE present.
- Bundle size ≤ 128KB.

**Dynamic checks (execute bundle end-to-end):**
- Bundle instantiates without crash.
- First tick emits > 0 orders (catches silent crashes).
- HYDROGEL orders have `symbol=HYDROGEL_PACK` and bids ≥ 9999 (catches quote-collapse).
- VEV_4000 orders have `symbol=VEV_4000`; no VELVET order has a price in ~1250 range (catches _PRODUCT swap).
- VELVET orders are EMPTY at pos=0, net_delta=0 (catches missing idle guard).
- Zero-bid lottery orders appear on VEV_6000 and VEV_6500.
- 20 sequential ticks run without exception and without runaway order counts.

Verified the **old** broken bundles fail these checks; the **new** bundle passes all 13.

## Open questions for next submission

1. **HYDROGEL fill rate on live platform.** Our LOO replay assumed 1.6% passive fill and predicted +$8.6K. On-platform HYDROGEL = 0 trades in both bad submissions. True platform fill rate unknown. The fix puts correct orders in the book; we'll see.

2. **Whether quote-inside-wall vs join-best is better for Prosperity.** Currently HYDROGEL quotes one tick INSIDE the deep wall, BEHIND the best bid. This means we only fill when the best-bid level is consumed. Alternative: join best-bid (quote AT 10003 not 10002). Would need an A/B on live data.

3. **Whether the first-bundle (non-minified) orchestrator.step drop is already fixed.** The static regression test passes because we only rebuild the minified bundle. If someone rebuilds without `--minify`, does it still drop the step() call? Needs investigation; for now, **always use `--minify`** when exporting.
