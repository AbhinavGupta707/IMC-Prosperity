## Read

Promote only variants that improve public full-day behavior and sliding-window
left tail, or that add official PnL with a clean mechanism and strong controls.
Official-only gains are calibration, not final-1M proof.

## Decision

Do **not** export a new VELVET upload from this stage.

The only higher official-proxy candidate was
`one_shot_core_rebound5262_tp10`, but it scored only `+37.0` over the matched
negative control on the official 100k proxy (`3276.5` vs `3239.5` over
`sell7_base`). That is not enough mechanism separation to justify an upload for
the final 1M objective. It is almost the same fragile core-recycler effect we
already saw in `probe_stack`.

The attempted rolling/session-open gates are worse:

- `session_drop40_v5248_5264` and `session_rebound40_r5_v5248_5264` activate
  on all three historical days and lose about `-6.5k` historical mean versus
  `sell7_base`.
- `session_drop40_plus80` is even worse at about `-11.3k` historical mean.
- They do not improve the official proxy versus the simpler delayed gate; they
  collapse back to the `+1,528.5` delayed-gate behavior on the official path.

Current final-1M VELVET read:

1. Keep `VEV_5500 sell7`.
2. Treat `probe_stack` as the max official-100k VELVET-complex calibration
   candidate, not as proven robust alpha.
3. Prefer `delayed_full` / `plus80` style exposure control only if we want to
   reduce terminal path risk.
4. Do not add this stage's core reduce-only or session-drop controllers.

Remaining VELVET alpha is not exhausted, but this one-go next-gen attempt did
not find a robust additional layer. The next credible path would need a more
selective regime classifier, likely using path shape and counterparty/liquidity
features, not another open-drawdown threshold.
